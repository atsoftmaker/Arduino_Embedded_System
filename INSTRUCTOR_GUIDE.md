**กลุ่มเป้าหมาย:** วิศวกร/ช่างเทคนิคของลูกค้า ที่ต้องนำระบบไปติดตั้งใช้งานจริงหน้างาน

**ขนาดกลุ่ม:** แนะนำ 2–3 คน/ชุดอุปกรณ์ (Lab RS485 ต้องใช้ ≥2 บอร์ดต่อกลุ่ม)

## 4. จุดที่ผู้เรียนมักติด (Common Pitfalls) + วิธีแก้

### พื้นฐาน / ภาษา C
- **อัปโหลดไม่ได้** → ติดตั้งไดรเวอร์ CH34x (มีใน repo Lab_Arduino_UNO), เลือก Port/Board ให้ถูก
- **Serial Monitor เป็นภาษาต่างดาว** → ตั้ง Baud Rate ให้ตรง (UNO=9600, ESP32=115200)
- ในไฟล์ `exercise12/13.cpp` (repo เดิม) มี `While` ตัว W ใหญ่ ซึ่งคอมไพล์ไม่ผ่าน — ของจริงต้องเป็น `while` ตัวเล็ก (ใช้เป็นแบบฝึกหา bug ได้)

### Relay / Control
- **Relay ทำงานกลับด้าน** → โมดูลส่วนใหญ่ Active LOW สลับ `RELAY_ON/OFF` ในโค้ด
- **LED หรี่ไม่ได้** → ต้องใช้ขา PWM (มีสัญลักษณ์ `~`: D3,5,6,9,10,11 บน UNO)

### RS485 / Modbus
- **ส่งแล้วไม่มีอะไรมา** → 90% คือ (1) ไขว้สาย A/B (2) ลืม GND ร่วม (3) ลืมสลับ DE/RE
- **ได้ข้อมูลขาดๆ** → ลืม `flush()` ก่อนสลับกลับมารับ
- **Modbus error/timeout** → Slave ID ไม่ตรง, Baud ไม่ตรง, หรือ `delay()` ใน loop ของ Slave นานเกินจน `poll()` ไม่ทัน

### ESP32 / Wi-Fi
- **บอร์ดเสียหลังต่อสาย** → ต่อ 5V เข้าขา GPIO โดยตรง (ย้ำซ้ำๆ เรื่อง 3.3V)
- **Enterprise เชื่อมไม่ติด** → ลองสลับรูปแบบ Username, เช็คบัญชีไม่ถูกล็อก, MAC อาจถูกบล็อก
- **คอมไพล์ WPA2_AUTH_PEAP ไม่ผ่าน** → ESP32 Core เก่า ให้ใช้วิธี `esp_wpa2.h` (มีในไฟล์ Lab 24)
- **POST ไป OPS ไม่สำเร็จ** → ESP32 ต้องอยู่วงเดียว/route ถึง 10.5.15.5, ตรวจ endpoint และ firewall

## 5. เฉลยแนวคิดคำถามท้าย Lab (สำหรับวิทยากร)

- **Lab 27** — Safety system ควรใช้ **NC** เพื่อให้ "ไฟดับ = เครื่องหยุด" (Fail-safe); ต่อโหลดที่ NO เพื่อให้ปกติวงจรเปิด รอสั่งงาน
- **Lab 15** — เพิ่มความละเอียดเกิน 10-bit ต้องใช้ ADC ภายนอกหรือ Oversampling; **ESP32 ADC = 12-bit (0–4095)**
- **Lab 16** — Open Loop เหมาะกับงานที่ผลลัพธ์คาดเดาได้แน่นอน เช่น ไฟกะพริบ, สายพานเดินตามเวลา
- **Lab 17** — Hysteresis = 0 ทำให้ Relay สลับถี่มาก (สึกหรอ); กว้างไป = ค่าแกว่งห่างเป้า ควบคุมหยาบ
- **Lab 18** — เทอม **I (Integral)** กำจัด Steady-State Error เพราะสะสม error ที่ค้างนานจนดันระบบถึงเป้า
- **Lab 19** — RS485 ใช้ Differential (A-B) จึงทนรบกวนและส่งไกล; RS232 ใช้ single-ended เทียบ GND ส่งใกล้
- **Lab 21** — Address ซ้ำ → 2 Slave ตอบพร้อมกัน เกิด Collision สัญญาณชน Master อ่านเพี้ยน
- **Lab 22** — CRC-16 ตรวจความถูกต้องของ Frame; ถ้าผิด Slave จะ **ทิ้ง Frame เงียบๆ** (ไม่ตอบ) Master จะ timeout แล้ว retry
- **Lab 24** — Hardcode รหัส AD เสี่ยงเพราะใครถอด flash ออกมาก็อ่านได้ และรหัสองค์กรมีสิทธิ์เข้าระบบอื่น; Lab 25 ย้ายไป NVS + ตั้งผ่านเว็บ
- **Lab 27** — OPS ล่ม ควรทำ **store-and-forward**: เก็บข้อมูลลง NVS/SD พร้อม timestamp แล้วส่งย้อนหลังเมื่อกลับมาออนไลน์

## 7. เชื่อมโยงสู่การติดตั้งจริง (สำหรับวันที่ 3 — ประชุมปิดงาน หลังอบรม 30 วัน)

ประเด็นที่ควรสรุปกับลูกค้า:
- Mapping จุดติดตั้งจริง: Node แต่ละตัววางที่ไหน, Gateway ครอบคลุมกี่ Node
- มาตรฐานการเดินสาย RS485 หน้างาน (ระยะ, Termination, การกราวด์)
- นโยบายบัญชี Wi-Fi สำหรับอุปกรณ์ (Service account vs บัญชีบุคคล) กับฝ่าย IT
- รูปแบบ API/Schema ของ OPS Server ที่จะใช้จริง และแผนการ scale จำนวนอุปกรณ์
- แผนสำรองข้อมูล (store-and-forward) และการมอนิเตอร์สถานะ Gateway

## ภาคผนวก A: Mock OPS Server (Python Flask) สำหรับทดสอบ Lab 27

ใช้รันบนโน้ตบุ๊กวิทยากรเพื่อจำลอง OPS Server ตอนอบรม (แก้ IP ในโค้ด ESP32 ให้ชี้มาที่ IP เครื่องนี้):

```python
# pip install flask
from flask import Flask, request
app = Flask(__name__)

@app.route('/api/data', methods=['POST'])
def data():
    print("DATA :", request.json)
    return {"status": "ok"}, 200

@app.route('/api/alert', methods=['POST'])
def alert():
    print("ALERT:", request.json)
    return {"status": "ok"}, 200

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=80)
```
รันด้วย `sudo python3 mock_ops.py` (พอร์ต 80 ต้องสิทธิ์ admin) แล้วดู log การส่งจาก ESP32 แบบเรียลไทม์
